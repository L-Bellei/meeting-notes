# Kanban Board Design

## Goal

Transform meetings into "stories" on a global kanban board, providing operational visibility over the progress of action items generated by each meeting.

## Context

The app already generates tasks automatically via AI after each meeting, but they have no lifecycle or progress visibility. This feature adds a board where each meeting becomes a card that moves through user-defined columns (Backlog → Em Andamento → Concluído by default).

## Core Decisions

- **One card per meeting** — tasks are a checklist inside the card, not board-level cards
- **Board is global** — single board, all themes, no per-theme board
- **Column configuration is global** — not per-theme
- **Card numbering is global** — `#1, #2, #3...` sequential and immutable; numbers are never reused when a card is deleted (issue-tracker style)
- **Auto-add is per-theme** — each theme decides if its meetings enter the board automatically after AI processing, or require a manual "Add to Board" action
- **Old meetings** — not migrated; they appear outside the board until the user adds them manually; they receive no retroactive numbers
- **Card updates on move** — moving a card updates `updated_at` and `status` (mirroring the column name); no configurable column rules (YAGNI)
- **Board navigation** — top-level destination alongside Meetings and Settings; not a toggle of the existing meeting list

## Card Design

Cards display (Option C):
- Number (`#12`), title, theme badge with color
- 2-line excerpt of the AI-generated summary
- Progress dots (filled = completed task, empty = pending)
- `created_at` relative timestamp

Clicking a card opens a full modal with: editable description, task checklist (checkboxes call existing task endpoints), full summary, key points.

## Schema

### Migration `007_board.sql`

```sql
ALTER TABLE themes ADD COLUMN auto_add_to_board INTEGER NOT NULL DEFAULT 0;

CREATE TABLE board_columns (
    id         TEXT PRIMARY KEY,
    name       TEXT NOT NULL,
    position   REAL NOT NULL,
    created_at DATETIME NOT NULL
);

INSERT INTO board_columns (id, name, position, created_at) VALUES
    ('col-backlog',    'Backlog',       1000, datetime('now')),
    ('col-wip',        'Em Andamento',  2000, datetime('now')),
    ('col-done',       'Concluído',     3000, datetime('now'));

CREATE TABLE board_cards (
    id          TEXT PRIMARY KEY,
    meeting_id  TEXT NOT NULL UNIQUE REFERENCES meetings(id) ON DELETE CASCADE,
    column_id   TEXT NOT NULL REFERENCES board_columns(id),
    number      INTEGER NOT NULL UNIQUE,
    position    REAL NOT NULL,
    description TEXT NOT NULL DEFAULT '',
    updated_at  DATETIME NOT NULL,
    created_at  DATETIME NOT NULL
);

CREATE INDEX idx_board_cards_column ON board_cards(column_id);
CREATE INDEX idx_board_cards_number ON board_cards(number);
```

**Notes:**
- `number` is generated as `SELECT COALESCE(MAX(number), 0) + 1 FROM board_cards` inside a transaction; safe for single-user SQLite
- `description` is pre-populated from `summaries.content` at card creation if a summary exists
- `position` is a float (REAL); insertion between two cards uses `(pos_a + pos_b) / 2`; rebalancing (renumber with step 1000) triggers when `|pos_b - pos_a| < 1e-9`
- Minimum 1 column enforced at the service layer; the last column cannot be deleted
- `ON DELETE CASCADE` on `meeting_id`: deleting a meeting removes its card

## API Endpoints

### Columns

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/board/columns` | List columns ordered by position, with card count each |
| POST | `/api/board/columns` | Create column |
| PUT | `/api/board/columns/{id}` | Rename column |
| DELETE | `/api/board/columns/{id}` | Delete; requires `?move_to={columnId}` if column has cards |
| PATCH | `/api/board/columns/reorder` | Reorder: `[{id, position}]` |

`DELETE` behavior:
- No cards → delete immediately
- Has cards, no `move_to` → 422 `{"cards_count": N, "message": "..."}`
- Has cards, `move_to` provided → transactional UPDATE + DELETE

### Cards

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/board/cards` | List all cards (denormalized: card + meeting + theme + task_progress) |
| POST | `/api/board/cards` | Add meeting to board; `{meeting_id, column_id?}` |
| GET | `/api/board/cards/{id}` | Full detail: card + meeting + summary + tasks + key_points |
| PUT | `/api/board/cards/{id}` | Update description |
| DELETE | `/api/board/cards/{id}` | Remove from board (does NOT delete the meeting) |
| PATCH | `/api/board/cards/{id}/move` | Drag-drop: `{column_id, position}`; updates `updated_at` |

Query params for `GET /api/board/cards`: `?title=&number=&created_after=&created_before=&updated_after=&updated_before=`

`GET /api/board/cards` returns one query with JOINs (no N+1); each item includes `task_progress: {total, completed}` and `status` (derived from `column.name` — not stored in the table).

`PATCH /api/board/columns/reorder` — chi routes literal segments before parameterized ones, so `/reorder` correctly takes priority over `/{id}`.

### Theme (extension)

`PUT /api/themes/{id}` — already exists; now also accepts `auto_add_to_board: bool`.

## Go Architecture

Follows existing patterns:

```
internal/
  database/migrations/007_board.sql
  models/models.go                      — add BoardColumn, BoardCard structs
  repository/
    board_column_repository.go          — List, Create, Update, Delete, Reorder
    board_card_repository.go            — List(filters), Create, GetByID, GetDetail,
                                          UpdateDescription, Move, Delete
  services/
    board_column_service.go             — CRUD + delete-with-move logic + min-1 guard
    board_card_service.go               — Create (with number gen + description pre-pop),
                                          Move (with position rebalancing), rest are thin
  handlers/
    board_handler.go                    — all board endpoints in one handler file
cmd/api/main.go                         — wire board handler
cmd/desktop/app.go                      — wire board handler
```

`Orchestrator.runAIGeneration` gains a step after task generation: if `meeting.ThemeID != nil` and `theme.AutoAddToBoard == true` and at least one column exists, call `boardCardSvc.Create(ctx, meetingID, "")`.

## Frontend Architecture

**New dependency:** `@dnd-kit/core @dnd-kit/sortable @dnd-kit/utilities`

```
frontend/src/
  hooks/
    useBoard.ts              — useColumns(), useCards(filters), useMoveCard(),
                               useCreateCard(), useDeleteCard(), useUpdateCard()
    useBoardColumns.ts       — useCreateColumn(), useUpdateColumn(),
                               useDeleteColumn(), useReorderColumns()
  components/board/
    BoardView.tsx            — DndContext wrapper, renders columns
    KanbanColumn.tsx         — column header (name + count) + sortable card list
    KanbanCard.tsx           — draggable card (summary excerpt + progress dots)
    CardDetailModal.tsx      — full detail: editable description, task checklist,
                               summary, key points
    BoardFilters.tsx         — title search, number filter, date range pickers
    ColumnSettingsPanel.tsx  — add/rename/reorder/delete columns
```

Navigation change in `App.tsx`: top bar gains `[Reuniões] [Board] [Configurações]`. `BoardView` replaces the full content area when active.

`MeetingDetail.tsx` gains a "Adicionar ao Board" button when the meeting has no card. After adding, shows `#N · Ver no Board` badge that navigates to BoardView.

**Drag-and-drop flow:**
1. Drop → optimistic update via `queryClient.setQueryData`
2. `PATCH /api/board/cards/{id}/move` fires in background
3. On error → `onError` callback reverts via `queryClient.invalidateQueries`
